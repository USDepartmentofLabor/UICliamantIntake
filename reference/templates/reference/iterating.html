Many state workforce agencies are working in challenging environments that include legacy technology. States have limited resources and workloads that can be unpredictable. These factors can make large scale modernization difficult, expensive, and/or time-consuming. However, there are opportunities to make incremental and impactful changes.

The case study below demonstrates how the state of New Jersey has successfully made significant improvements despite their current technical challenges.

Go mobile-friendly with existing technology

Mobile-friendly systems and websites are critical given that smart phones are the most accessible (and often only) for many applicants. Few states have mobile-friendly experiences today. According to a Pew Research study titled, “Racial and ethnic differences in how people use mobile technology”, 25% of minority populations have smartphones as their only internet device.

Using cascading style sheets (CSS) media queries, you can make huge improvements to the mobile-friendliness of your site, often without the need to upgrade your software stack or infrastructure. Media query example

The following code snippet shows a couple sections of text aligned side-by-side on a page using the CSS flex display property. This works well for wide screens, but appears too narrow on mobile. 

<style>
  .container {
    display: flex;
    gap: 1rem;
  }
  .container div {
    background-color: #eee;
    padding: 1rem;
    flex-grow: 1;
  }
</style>
<div class="container">
  <div>
    Lorem ipsum dolor sit amet, consectetur adipiscing elit, sed do eiusmod
    tempor incididunt ut labore et dolore magna aliqua.
  </div>
  <div>
    Lorem ipsum dolor sit amet, consectetur adipiscing elit, sed do eiusmod
    tempor incididunt ut labore et dolore magna aliqua.
  </div>
</div>

To fix this, we can add a media query. The added code in the following snippet translates to "apply these styles if the viewport width is less than or equal to 480 pixels." In this case, we change the container from display: flex to display: block and add some margin around the container's children. 

<style>
  .container {
    display: flex;
    gap: 1rem;
  }
  .container div {
    background-color: #eee;
    padding: 1rem;
    flex-grow: 1;
  }
  /* Added media query */
  @media (max-width: 480px) {
    .container {
      display: block;
    }
    .container div {
      margin: 1rem;
    }
  }
</style>
<div class="container">
  <div>
    Lorem ipsum dolor sit amet, consectetur adipiscing elit, sed do eiusmod
    tempor incididunt ut labore et dolore magna aliqua.
  </div>
  <div>
    Lorem ipsum dolor sit amet, consectetur adipiscing elit, sed do eiusmod
    tempor incididunt ut labore et dolore magna aliqua.
  </div>
</div>

With this change in hand, we can adjust our screen size and see that, when we cross the 480px breakpoint, our layout changes to be more mobile-friendly. For the example below, the mobile version automatically vertically stacks content to be mobile-friendly.

Experiment from scratch

When you are not constrained by legacy systems and process, you can make fast progress with new capabilities. You may uncover ideas that you otherwise wouldn't have.

New Jersey is not releasing this application as their new system, but are using pieces of it to improve their existing system. Change the frontend without requiring changes on the backend

A challenge to making changes to application form questions is that it could require changes to the database (or backend). But, it may be possible to update the applicant-facing frontend of the application without changing the backend. This is done by mapping new field options to existing options.

By changing the frontend form responses and developing a mapping schema, you can still conform to the database requirements. This enables changes on form questions to be made without a significant tech overhaul.

Make space to hear ideas

New Jersey, like many states and territories, has struggled to update their UI application because of complicated legacy technology problems. During the pilot, they brought together policy, IT, and innovation teammates to meet and brainstorm. They came up with technical workarounds that enabled them to make short term improvements.

Below are some of the tactics they used to create space to hear and prioritize ideas. If you haven't already, you can try some of these tactics in your organization to see if they result in any new learnings or opportunities.

  • They structured a meeting around an agenda like this:
    • What are the program office's goals? Define specific claimant-experience goals such as plain language for application questions, accessibility, or mobile-responsiveness.
    • What are the blockers? Discuss: are there too many other things prioritized, that are preventing action on the aforementioned goals? Is there a technology blocker? Are there other problems that make it difficult to make improvements?
    • What are some ideas for workarounds? Discuss how to solve the problems or change incrementally to reach claimant experience goals.
  • They made sure policy, IT, and innovation decision-makers were present at the meeting, so they could advocate for the importance of the claimant experience goals. Leaders selected ideas to pursue, and the teams scheduled follow-up sessions to dive into implementation details so that tactical logistics did not interfere with the brainstorming meeting.
  • They used a round-table style to facilitate the meeting so anyone from the technology side had the opportunity to speak, not just leaders. It specifically was important that the technical teammates had the space to propose ideas based on the program office's goals.
